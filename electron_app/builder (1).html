⚛ Site → App (React Native)
Enveloppe un site web dans une vraie app React Native (react-native-webview). Méthode la plus fragile des trois modes hybrides : `react-native init` télécharge un template complet + installe des centaines de paquets npm, donc sensible à un réseau lent ou instable — prévois plusieurs minutes au 1er lancement. Nécessite les composants jdk, androidSdk, nodejs et reactNativeCli installés (onglet ⚙ Composants).
Extrait dans android/app/src/main/assets/www/ — l'app n'a besoin d'aucune connexion réseau pour s'ouvrir.
Le package Java et les dépendances déjà présentes ne sont jamais modifiés — seuls les champs ci-dessous sont surchargés, et uniquement si tu les remplis. Si node_modules/ est absent du zip, un `npm install` est relancé automatiquement.
Redimensionnée automatiquement pour toutes les densités Android (l'icône adaptative du template est désactivée au profit du PNG classique). Icône bleue par défaut si non fournie.

🔍 Décompiler un APK (jadx)
Transforme le bytecode d'un APK en sources Java lisibles pour l'analyser (audit, debug, inspiration). Nécessite le composant jadx installé (onglet ⚙ Composants).
⚠ Limites à connaître
  • Code natif (.so, C/C++) — jadx ne décompile pas ces bibliothèques en Java, il les ignore.
  • APK obfusqués/protégés (ProGuard agressif, R8, packers commerciaux type DexGuard) — sources partielles ou illisibles, parfois échec total. Aucun outil ne peut garantir de contourner ça.
  • App Bundles (.aab) — pris en charge : convertis automatiquement en APK universel (bundletool) avant décompilation.
  • APK split isolé (config split langue/densité/ABI, sans classes.dex) — pas de code à décompiler ; utilise plutôt le split base.apk.
  • APK corrompus ou classes.dex invalide — échec propre : l'erreur remonte dans le journal au lieu de planter en silence.
